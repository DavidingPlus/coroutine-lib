# 定时器与传统定时器的区别

学习时间轮、时间堆等数据结构时，我发现它们与自己原来理解的"定时器"并不完全一样。

我以前理解的定时器更像是：

> **到时间以后，由操作系统主动发送一个信号或中断通知程序。**

例如：

```text
sleep(5)

↓

5 秒以后

↓

操作系统唤醒当前线程
```

或者：

```text
setitimer()

↓

5 秒以后

↓

收到 SIGALRM
```

这里的特点是：

- 时间到了以后，系统主动通知应用程序。
- 应用程序无需自己检查时间。
- 是否触发由操作系统保证。

因此，它更像是一种**事件通知机制（Event Notification）**。

---

## 本文中的定时器是什么？

而时间轮、时间堆、升序链表等实现，本质上并不是"主动通知"。

它们更准确地说，是：

> **一种超时任务管理器（Timeout Manager）。**

例如：

```cpp
addTimer(5000, callback);
```

这里只是告诉定时器管理器：

> "请帮我记录这个任务，它将在 5 秒后超时。"

真正什么时候执行，还需要有人不断调用：

```cpp
tick();
```

或者：

```cpp
listExpiredCallbacks();
```

去检查：

```text
当前时间 >= 超时时间？
```

如果已经超时：

```text
↓

执行 callback
```

否则：

```text
↓

继续等待
```

因此，它本质上是一种：

> **轮询（Polling）检查超时。**

---

## 为什么会有延迟？

由于需要主动检查，因此：

```text
真正执行时间

=

超时时间

+

下一次检查时间
```

例如：

```text
超时时间：

10:00:00
```

但是：

```text
tick()

每 100ms 调用一次
```

那么：

可能出现：

```text
10:00:00.085

才真正执行
```

因此：

时间轮、时间堆等定时器都存在：

> **检查周期（Tick）带来的调度延迟。**

Tick 越小：

- 定时越精确；
- CPU 开销越大。

Tick 越大：

- CPU 开销越小；
- 延迟越明显。

这也是时间轮设计中需要权衡的重要因素。

---

## 为什么大家仍然使用这种方案？

既然需要自己检查，为什么不用操作系统的定时器？

原因是：

操作系统定时器适合管理**少量高精度定时任务**。

而对于网络服务器而言，通常需要管理：

```text
几十万

甚至

上百万
```

个连接超时。

例如：

```text
TCP KeepAlive

HTTP 长连接

RPC 请求超时

心跳检测

Session 超时
```

如果每个连接都创建一个操作系统定时器：

- 内核维护成本高；
- 系统调用开销大；
- 可扩展性较差。

因此，大多数高性能服务器都会采用：

- 升序链表
- 时间堆（最小堆）
- 时间轮（Time Wheel）

等数据结构，在**用户态统一管理所有超时任务**。

---

## 谁负责调用 tick()？

很多初学者还有一个疑问：

> **既然不会自动触发，那么是谁一直调用 tick()？**

答案是：

通常由框架的 **Event Loop（事件循环）** 或 **Scheduler（调度器）** 定期调用。

例如：

```text
while (true)
{
    epoll_wait(...);

    timer.tick();

    处理网络事件...
}
```

也就是说：

整个服务器在不断循环：

```text
等待 IO

↓

检查定时器

↓

执行超时任务

↓

继续等待 IO
```

因此：

**真正驱动定时器运行的，不是定时器本身，而是整个事件循环（Event Loop）。**

---

## 总结

传统意义上的定时器：

> **到时间后，由操作系统主动通知应用程序。**

而本文介绍的升序链表、时间堆、时间轮等"定时器"：

> **本质上是一种超时任务管理器（Timeout Manager），负责管理大量待执行的超时任务，并依赖事件循环主动检查哪些任务已经到期。**

因此，它们更准确地说是：

> **用于高性能服务器的超时管理数据结构，而不是操作系统意义上的硬件/软件定时器。**

# 单层时间轮定时器（Single-Level Time Wheel）

## 一、简介

时间轮（Time Wheel）是一种高效的定时器管理数据结构，最早由 Varghese 和 Lauck 在论文 **Hashed and Hierarchical Timing Wheels** 中提出，目前广泛应用于网络服务器、RPC 框架和消息队列等场景。

与升序链表、最小堆等维护全局有序性的方案不同，时间轮将时间划分为固定大小的时间片（Tick），使用一个**环形数组（Circular Array）**模拟时间流逝，每个数组元素称为一个**槽（Slot）**。

时间轮中的定时器不会按照超时时间排序，而是根据超时时间计算其所在的槽（Slot）和剩余需要等待的圈数（Round）。当时间轮转动到对应槽时，再根据 Round 判断定时器是否真正到期。

因此，时间轮能够以 **O(1)** 的时间复杂度完成定时器的插入，非常适合管理大量定时任务。

# 二、时间轮结构

假设时间轮共有 **60 个槽**：

```text
                [0]
           /             \
       [59]               [1]
      /                     \
    [58]                     [2]
     |                        |
    ...                      ...
      \                     /
       [31] ... [30] ... [3]
```

每个槽表示固定的时间单位，例如：

- 一个槽 = 1 秒（1 Tick）
- 整个时间轮 = 60 秒

时间轮维护一个当前槽指针：

```cpp
current = 0;
```

每调用一次 `tick()`：

```cpp
current = (current + 1) % slotCount;
```

表示时间向前推进 **一个 Tick**，随后处理当前槽中的所有定时器。

> **说明：**
>
> 本实现约定：
>
> `timeout >= 1`
>
> 表示**经过 `timeout` 个 Tick 后执行定时器**。

# 三、定时器如何存放

例如：

当前：

```text
current = 10
```

添加一个：

```text
15 Tick 后执行
```

计算目标槽：

```cpp
index = (current + timeout) % slotCount
      = (10 + 15) % 60
      = 25
```

因此，将该定时器放入：

```text
slot[25]
```

中。

整个过程无需遍历，也无需排序，因此插入复杂度为 **O(1)**。

# 四、为什么需要 Round

如果：

```text
timeout = 80 Tick
```

而时间轮只有：

```text
60 个槽
```

则：

```cpp
index = (10 + 80) % 60
      = 30
```

如果仅记录槽的位置，那么时间轮第一次转到：

```text
slot[30]
```

时就会错误执行。

因此，需要额外记录：

```cpp
round = (timeout - 1) / slotCount
      = (80 - 1) / 60
      = 1
```

表示：

> **第一次经过该槽时不能执行，还需要等待一整圈；第二次经过该槽时才真正执行。**

Timer 定义如下：

```cpp
struct Timer
{
    uint32_t round = 0;

    std::function<void()> callback;
};
```

其中：

- `round`：剩余需要等待的完整圈数。
- `callback`：定时器超时后的回调函数。

# 五、Tick 工作流程

每调用一次：

```cpp
tick();
```

时间轮完成以下两个步骤：

```text
① current 向前移动一个槽

↓

② 检查当前槽中的所有定时器
```

对于每个 Timer：

```text
round > 0
    ↓
round--

round == 0
    ↓
执行 callback
    ↓
删除 Timer
```

例如：

```text
slot[20]

↓

TimerA  round = 0

TimerB  round = 2

TimerC  round = 1
```

当时间轮转动到该槽时：

```text
TimerA
↓

执行并删除

TimerB
↓

round = 1

TimerC
↓

round = 0
```

下一次时间轮再次转动到：

```text
slot[20]
```

时：

```text
TimerC
↓

执行
```

再下一圈：

```text
slot[20]
```

时：

```text
TimerB
↓

执行
```

整个过程中，仅处理当前槽中的定时器，而不会遍历整个时间轮。

# 六、为什么时间轮能够做到 O(1)

时间轮最大的优势就是**插入效率高**。

对于升序链表、最小堆等结构，插入一个新的定时器时，都需要维护全局有序性，因此需要寻找插入位置：

- 升序链表需要遍历链表寻找位置。
- 最小堆需要进行上浮操作维持堆序。

而时间轮完全不同。

当添加一个新的定时器时，只需要根据超时时间计算：

```cpp
index = (current + timeout) % slotCount;

round = (timeout - 1) / slotCount;
```

随后直接插入对应槽即可。

整个过程：

- 不需要比较已有定时器。
- 不需要维护全局排序。
- 不需要移动其它节点。

因此，时间轮插入复杂度始终为：

```text
O(1)
```

这也是时间轮能够管理大量定时任务的根本原因。

# 七、复杂度分析

| 操作 | 时间复杂度 |
|------|-----------:|
| 添加定时器 | O(1) |
| 删除定时器 | O(1) |
| Tick | O(k) |

其中：

```text
k = 当前槽中的定时器数量
```

由于无需维护全局有序性，因此插入定时器仅需计算一次槽的位置即可完成。

# 八、优缺点

## 优点

- 添加定时器复杂度为 **O(1)**。
- 删除定时器复杂度为 **O(1)**。
- 无需维护全局排序。
- 非常适合管理大量定时任务。
- 数据按槽组织，局部性较好，实现简单。

## 缺点

- 时间精度受 Tick 大小限制。
- Tick 越小，时间轮越大，占用内存越多。
- Tick 越大，定时精度越低。
- 单层时间轮适合处理中短时间定时任务，不适合跨度特别大的超时。

因此，工业实现通常采用：

- 多层时间轮（Hierarchical Time Wheel）
- 哈希时间轮（Hashed Timing Wheel）

来支持更大的时间跨度。

# 九、与升序链表比较

| 项目 | 升序链表 | 单层时间轮 |
|------|---------:|-----------:|
| 插入 | O(n) | O(1) |
| 删除 | O(1) | O(1) |
| Tick | O(1)（检查链表头） | O(k) |
| 是否维护全局排序 | 是 | 否 |
| 是否需要寻找插入位置 | 是 | 否 |

其中：

- **升序链表**适合定时器数量较少、超时时间变化频繁的场景。
- **单层时间轮**适合网络服务器、连接管理等拥有大量定时任务的场景。

# 十、多层时间轮（Hierarchical Time Wheel）

## 为什么需要多层时间轮？

单层时间轮虽然具有 **O(1)** 的插入效率，但它有一个明显的问题：

> **时间跨度有限。**

例如，一个时间轮共有 **60 个槽**，每个槽表示 **1 秒**：

```text
1 Tick = 1 秒

60 个槽
```

整个时间轮只能表示：

```text
60 秒
```

如果需要添加一个：

```text
3 小时后执行
```

的定时器，那么：

```text
3 小时 = 10800 秒
```

对应：

```cpp
round = 10800 / 60 = 180
```

意味着：

每当时间轮转动一圈，都需要：

```text
round--
```

直到：

```text
round == 0
```

才能真正执行该定时器。

虽然算法复杂度仍然是 **O(1)**，但对于大量长时间定时器而言，需要维护大量 `round`，效率并不理想。

因此，引入了**多层时间轮**。

# 十一、多层时间轮的思想

多层时间轮可以理解为：

> **使用多个不同时间粒度的时间轮，共同管理定时器。**

其思想与现实中的时钟完全一致：

```text
        天轮
         │
         ▼
        时轮
         │
         ▼
        分轮
         │
         ▼
        秒轮
```

不同层表示不同的时间粒度：

| 时间轮 | 一个槽表示 |
|---------|-----------|
| 秒轮 | 1 秒 |
| 分轮 | 1 分钟 |
| 时轮 | 1 小时 |
| 天轮 | 1 天 |

越高层表示的时间跨度越大。

# 十二、定时器如何存放？

例如：

当前时间：

```text
00:00:00
```

需要添加一个：

```text
2 小时 30 分 20 秒后执行
```

如果使用单层时间轮，则需要维护一个非常大的 `round`。

而多层时间轮会直接将其放入：

```text
时轮

↓

slot[2]
```

表示：

```text
还有 2 小时。
```

此时：

秒轮、分轮中都不会保存该定时器。

# 十三、什么是 Cascade（级联）？

多层时间轮最核心的思想就是：

> **定时器会随着时间逐层下降。**

例如：

```text
2 小时

30 分

20 秒
```

最开始：

```text
时轮

↓

slot[2]
```

两个小时后：

```text
时轮

↓

slot[2]

↓

Timer
```

此时，并不会立即执行。

而是重新计算剩余时间：

```text
30 分

20 秒
```

随后：

放入：

```text
分轮

↓

slot[30]
```

再经过 30 分钟：

继续下降：

```text
秒轮

↓

slot[20]
```

最后：

20 秒以后：

```text
执行 callback
```

整个生命周期如下：

```text
时轮

↓

分轮

↓

秒轮

↓

执行
```

这个过程称为：

- Cascade（级联）
- 或者 Timer Migration（定时器迁移）

# 十四、多层时间轮为什么更高效？

假设系统中存在：

```text
100 万个定时器
```

其中：

```text
99%

都是几小时以后才触发。
```

如果采用单层时间轮：

所有定时器都会存在同一个时间轮中，仅依靠 `round` 区分。

而多层时间轮中：

```text
小时以后

↓

时轮

几分钟以后

↓

分轮

几十秒以内

↓

秒轮
```

真正位于秒轮中的定时器数量通常非常少。

因此：

- 秒轮每 Tick 处理的数据更少。
- 长时间定时器无需频繁减少 `round`。
- 更适合管理海量连接和定时任务。

# 十五、多层时间轮与单层时间轮比较

| 项目 | 单层时间轮 | 多层时间轮 |
|------|-----------|-----------|
| 插入 | O(1) | O(1) |
| 删除 | O(1) | O(1) |
| Tick | O(k) | O(k) |
| 时间跨度 | 较小 | 非常大 |
| 是否需要大量 Round | 是 | 否 |
| 实现复杂度 | 简单 | 较高 |

其中：

- **单层时间轮**通过 **Slot + Round** 管理未来时间。
- **多层时间轮**通过 **多个不同粒度的时间轮**管理未来时间，仅在临近超时时逐层下放定时器。

因此，多层时间轮可以看作是：

> **将单层时间轮中的大量 Round，转换成多个不同时间粒度的时间轮，从而减少长时间定时器的维护开销。单层时间轮的核心思想是「Slot + Round」；多层时间轮则进一步将「Round」拆分到多个不同时间粒度的时间轮中，通过定时器的逐层迁移（Cascade）实现对超长时间定时任务的高效管理。**

# 十六、Linux 中的多层时间轮

Linux 早期内核（4.x 之前）采用的就是经典的多层时间轮（Hierarchical Timing Wheel）。

其结构大致如下：

```text
TVR（256 个槽）

        │
        ▼

TVN1（64 个槽）

        │
        ▼

TVN2（64 个槽）

        │
        ▼

TVN3（64 个槽）

        │
        ▼

TVN4（64 个槽）
```

其中：

- TVR（Timer Vector Root）负责管理最近即将到期的定时器。
- TVN1 ~ TVN4 负责管理越来越远的未来时间。

随着时间推进，定时器会不断从高层时间轮迁移到低层时间轮，直到进入最底层时间轮后最终执行。

这种设计能够以较低的时间复杂度管理海量定时器，因此曾长期应用于 Linux 内核的定时器管理。

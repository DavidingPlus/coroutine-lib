# 升序链表定时器（Sorted Linked List Timer）

## 一、简介

升序链表（Sorted Linked List）是一种经典的定时器实现方案，也是许多网络编程教材介绍定时器时最先使用的数据结构，例如《Linux 高性能服务器编程》第 11 章。

它的核心思想非常简单：

> **始终按照超时时间从小到大维护一个有序双向链表。**

链表头永远都是**最先超时**的定时器，因此每次检查是否有定时任务到期时，只需要检查链表头即可。

由于实现简单、逻辑清晰，因此升序链表非常适合作为学习定时器原理的第一种实现方式。

## 二、数据结构

每一个定时器保存：

```cpp
struct Timer
{
    uint64_t expire;

    std::function<void()> callback;

    Timer* prev;
    Timer* next;
};
```

其中：

- expire：定时器超时时间（毫秒）
- callback：超时后执行的回调函数
- prev / next：双向链表指针

TimerManager 维护整个链表：

```text
Head
 │
 ▼
1000

↓

3000

↓

5000

↓

7000
```

链表始终按照：

```text
expire
```

升序排列。

因此：

> Head 永远表示最近即将超时的定时器。

## 三、添加定时器

添加新的 Timer 时，需要按照 expire 的大小寻找插入位置。

例如：

当前链表：

```text
1000

↓

3000

↓

5000
```

插入：

```text
2000
```

最终：

```text
1000

↓

2000

↓

3000

↓

5000
```

因此：

添加定时器需要遍历链表寻找插入位置。

时间复杂度：

```text
O(n)
```

## 四、删除定时器

删除操作比较简单。

例如：

```text
1000

↓

2000

↓

3000
```

删除：

```text
2000
```

结果：

```text
1000

↓

3000
```

由于采用双向链表，只需要修改前后两个节点即可。

时间复杂度：

```text
O(1)
```

（前提是已经拿到了 Timer 节点。）

## 五、调整定时器

很多网络程序都会出现：

```text
客户端持续发送数据

↓

重新计算超时时间
```

例如：

连接原本：

```text
30 秒超时
```

收到新的数据后：

```text
重新开始计时

↓

30 秒后再超时
```

因此：

需要修改 Timer 的 expire。

本项目采用最简单的方法：

```text
删除 Timer

↓

修改 expire

↓

重新插入链表
```

也就是：

```cpp
delTimer(timer);

timer->expire = newExpire;

addTimer(timer);
```

虽然不是最优实现，但是逻辑最容易理解。

## 六、Tick 工作流程

TimerManager 不会自动检测时间。

外部需要周期性调用：

```cpp
tick();
```

例如：

```cpp
while (true)
{
    manager.tick();

    sleep(100ms);
}
```

tick() 的工作流程如下：

```text
检查链表头

↓

已经超时？

↓

执行回调

↓

删除节点

↓

继续检查新的链表头

↓

直到遇到第一个没有超时的 Timer
```

例如：

当前：

```text
now = 3500
```

链表：

```text
1000

↓

2000

↓

3000

↓

5000
```

tick() 会依次执行：

```text
1000

↓

2000

↓

3000
```

最后：

链表变成：

```text
5000
```

由于链表已经有序，因此：

一旦发现：

```text
Head

没有超时
```

后面的所有 Timer 一定也没有超时。

因此可以立即结束遍历。

## 为什么 Tick 只需要检查链表头？

由于整个链表始终按照超时时间升序排列：

```text
1000

↓

3000

↓

5000

↓

7000
```

如果当前时间：

```text
now = 2500
```

那么：

```text
1000
```

已经超时。

继续检查：

```text
3000
```

发现：

```text
3000 > 2500
```

说明：

```text
3000
```

没有超时。

由于链表已经有序，因此：

```text
5000

7000
```

一定也没有超时。

因此：

一旦遇到第一个未超时的节点，就可以立即停止遍历。

这也是升序链表相比普通链表最大的优势之一。

## 七、复杂度分析

| 操作 | 时间复杂度 |
|------|-----------|
| 添加定时器 | O(n) |
| 删除定时器 | O(1) |
| 修改定时器 | O(n) |
| Tick | O(k) |

其中：

```text
k
```

表示：

本次已经超时的定时器数量。

由于链表已经有序，

Tick 并不会遍历整个链表，而是只处理已经超时的部分。

## 八、优缺点

### 优点

- 实现简单
- 容易理解
- Tick 效率较高
- 适合定时器数量较少的场景

### 缺点

- 插入效率较低
- 调整超时时间需要重新排序
- 当定时器数量很多时，维护有序链表成本较高

因此：

随着定时器数量增加，

通常会采用：

- 最小堆（Min Heap）
- 时间轮（Time Wheel）
- 分层时间轮（Hierarchical Time Wheel）

等更加高效的数据结构。
